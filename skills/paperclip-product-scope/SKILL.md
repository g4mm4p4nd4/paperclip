# Paperclip Product Scope

Use this skill when the problem is still too big, too fuzzy, or too solution-shaped.

## When to Use

- the request mixes strategy, product, design, and engineering concerns
- the implementation path is obvious but the user outcome is not
- the work needs to be cut into phases, milestones, or validation steps
- the team keeps proposing parts instead of an end-to-end result

## Workflow

1. Define the user-visible outcome.
2. State the hypothesis being tested.
3. Identify the smallest complete version that still delivers value.
4. Split the work into atomic chunks with clear owners.
5. Separate must-have, nice-to-have, and future work.
6. Surface blockers, dependencies, and unknowns early.

## Rules

- Optimize for learning and shippability, not feature count.
- Prefer complete narrow slices over disconnected partial infrastructure.
- If a task crosses departments, break it into explicit child tasks.
- If the scope cannot fit the current appetite, cut work rather than extending silently.

## Output Expectations

Good output from this skill should leave behind:

- a clear problem statement
- a concrete success condition
- a phased or atomic plan
- explicit cuts
- named owners or target roles
